# Docs Agent Runner Configs

This directory holds the **editorial content** that controls how `Automattic/docs-agent` writes user-facing documentation for the Extra Chill network. It is consumed by the `docs` agent execution mode registered in `inc/docs-agent/docs-mode.php`.

## Files

- **`writing-rules.md`** — the editorial voice rules from extrachill-docs issue #6. Read verbatim by the `datamachine_agent_mode_docs` filter callback and injected into the AI's system context whenever an AI step runs with `agent_modes: ['docs']`. Edit this file and the next docs-agent run picks up the change automatically.
- **`platform-map.yml`** — per-plugin-repo platform identity (platform name, parent slug on docs.extrachill.com, audience description). Reserved for the per-target context layer (still pending — see "Per-target context" below).
- **`prompt-template.md`** — scaffold preserved from an earlier design iteration. Currently unused. May be repurposed as a per-target context template once the active-workspace exposure lands in Data Machine Code.

## How the integration actually works

The pattern mirrors `data-machine-editor`'s `editor` mode registration — a downstream plugin owns a Data Machine execution mode and supplies its guidance via filter.

```text
inc/docs-agent/docs-mode.php (this plugin)
        │
        │ registers 'docs' mode with Data Machine
        │ supplies writing-rules.md content via
        │ the datamachine_agent_mode_docs filter
        │
        ▼
Data Machine's AgentModeDirective (priority 22)
        │
        │ picks up the registered guidance whenever an
        │ AI step runs with agent_modes: ['docs']
        │
        ▼
AI system context
        │
        │ writing rules sit at priority 22, before
        │ pipeline memory (40), flow memory (45), and
        │ pipeline system_prompt (50)
        │
        ▼
docs-agent in CI generates user-facing markdown
that respects the editorial voice
```

No YAML wrapper. No custom directive class. No string mashing in caller workflows. The plugin registers a mode, supplies its guidance, and Data Machine handles the rest through its existing primitive.

## How a plugin repo opts in

Each plugin repo that wants user-facing docs generated by `Automattic/docs-agent` does two things:

1. **Add this plugin as a CI validation dependency** so it loads in the ephemeral WordPress runtime that drives the docs-agent run:
   ```yaml
   validation_dependencies: >-
     Automattic/agents-api@main,
     Extra-Chill/data-machine@main,
     Extra-Chill/data-machine-code@main,
     Extra-Chill/extrachill-docs@main,
     WordPress/ai-provider-for-openai@main
   ```

2. **Ensure the docs-agent bundle's user-docs pipeline / flow step has `agent_modes: ['docs']`** set in its step config so Data Machine activates the docs mode for that run.

The bundle-side change to add the mode is a small upstream patch to `Automattic/docs-agent`. Without it, docs.extrachill.com voice rules are not injected and the agent reverts to generic user-doc behavior.

## Editing the rules

When the voice rules change, edit `writing-rules.md` and commit. The next docs-agent CI run anywhere on the network picks up the change because `extrachill-docs` is pulled fresh per-run.

## Per-target context (pending)

Voice rules are network-wide and don't need per-repo customization. Per-target context — telling the agent which specific platform it's documenting (Artist Platform, Community, Newsletter, etc.) and who the audience is — does.

That layer requires the AI runtime to know which repo it's running against. Data Machine Code already tracks this in worktree metadata but does not yet expose it to AI execution payloads. Issue tracked separately (see extrachill-docs#33 thread). When that exposure lands, a second callback on `datamachine_agent_mode_docs` reads `platform-map.yml` and stacks per-target context onto the same mode at a later filter priority.

Until then, all docs-agent runs use the platform-wide voice rules only.

## Why these files live here, not in the docs-agent bundle

`Automattic/docs-agent` is a generic source-code documentation agent. Its `user-docs` flows ship sensible defaults for general-purpose user-facing documentation. The Extra Chill writing rules (no plugin / hook / slug naming, sixth-grade reading level, benefit-focused) are far stricter than the generic defaults and would be wrong to upstream.

This directory is where Extra Chill's editorial voice lives, versioned with the docs site itself. It travels with the plugin so it ships, deploys, and updates alongside docs.extrachill.com.
